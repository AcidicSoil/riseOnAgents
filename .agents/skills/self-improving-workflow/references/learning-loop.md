# Learning Loop — Episodic → Semantic → Rules

## Three-Layer Memory

```
.claude/memory/
├── episodic/              ← raw records, 1 file per event (gitignored)
├── semantic-patterns.json ← aggregated patterns (git-tracked)
└── working/               ← session-scope cache (gitignored)

.claude/rules/
└── dev-lessons.md          ← crystallized rules (auto-loaded by Claude)
```

**Episodic layer**: raw, fine-grained records of individual learning events. Written by reviewers when a `lessons_candidate` is present in their output. Gitignored — private to the machine and session.

**Semantic layer**: aggregated patterns across episodic records. One JSON file, git-tracked so the team shares accumulated pattern data. Each pattern tracks occurrence count, average confidence, and whether it has been promoted to a rule.

**Rules layer**: `dev-lessons.md` is the crystallized output — a human-readable, append-only file that Claude Code automatically loads at session start. Once a pattern is here, it is enforced.

---

## Episodic Record Format

Each episodic event is a single JSON file written to `.claude/memory/episodic/`:

```json
{
  "id": "ep-20260407-1142",
  "ts": "2026-04-07T11:42:00Z",
  "plan_id": "plan-20260407-001",
  "scope": "P1-S2-T3",
  "source": "implementation-reviewer",
  "category": "boundary|logic|spec|integration|style|risk|process",
  "what": "Short description of the pattern observed",
  "why": "Why this matters / what went wrong",
  "fix": "What the correct approach is",
  "fingerprint": "<category>:<sub>:<area>",
  "confidence": 0.85
}
```

Key field notes:
- `fingerprint`: generated by the writing reviewer at record time, not inferred by LLM during crystallization. Format is `<category>:<subcategory>:<area>` (e.g., `boundary:null-check:search`). The first 2 segments are used as the grouping key during crystallization.
- `confidence`: reviewer's confidence that this is a generalizable pattern (0.0–1.0).
- `source`: which reviewer wrote the record.

---

## Crystallization Algorithm

`scripts/crystallize.sh` runs the promotion logic deterministically:

```
for each new episodic E since last run:
  key = first 2 segments of E.fingerprint
  upsert pattern[key] in semantic-patterns.json:
    occurrences++
    running avg confidence updated
    append episodic_id to episodic_ids[]
    update last_seen timestamp

for each pattern P in semantic-patterns.json where promoted_to_rule = false:
  if P.occurrences >= 3 AND P.avg_confidence >= 0.7:
    append rule entry to dev-lessons.md
    set P.promoted_to_rule = true
    write decisions.jsonl entry: kind=learn, scope=global
```

---

## Hard-Coded Thresholds

**These thresholds are not user-tunable:**

- Minimum occurrences: **≥ 3**
- Minimum average confidence: **≥ 0.7**

Both conditions must be met simultaneously. A high-confidence single event does not promote. A low-confidence frequent pattern does not promote. The thresholds are intentionally conservative to prevent noise from polluting `dev-lessons.md`.

---

## dev-lessons.md Write Template

When crystallization promotes a pattern, it appends using this exact template:

```markdown
## L-{id}: {title}

**Rule**: ...

**Why**: ...

**How to apply**: ...

<!-- Crystallized: {date} | pattern: {fingerprint} | from {N} episodics | confidence: {avg} -->
```

`dev-lessons.md` is **append-only**. If a later pattern supersedes an earlier rule, the earlier entry is marked `⚠ superseded by L-{new}` — it is never deleted automatically. The full history is preserved.

---

## Triggers

Crystallization runs automatically at these points:

| Trigger | Who runs it |
|---|---|
| Each phase `done` | `/run` main loop calls `crystallize.sh` |
| `/run` exit (plan `done` or `blocked`) | `/run` main loop calls `crystallize.sh` |
| `/learn` manual command | User-initiated, runs anytime |

The episodic write (reviewer → `.claude/memory/episodic/`) happens immediately when a reviewer outputs a `lessons_candidate`. Crystallization is batched — it processes all new episodic records since the last run, not one at a time.
